# Архитектура vKOROBKU

## Компоненты

### vKOROBKU.App

WPF-клиент на .NET 8 с MVVM. Отвечает за библиотеку игр, выбор режимов, визуализацию анализа, ход операций и историю. Обычный запуск — `asInvoker`.

### Сканеры

- `SteamLibraryScanner` читает реестр, `libraryfolders.vdf` и `appmanifest_*.acf`;
- ручной источник принимает выбранный каталог, определяет игру по основному GOG ID, `steam_appid.txt`, EXE, имени папки и каталогу Steam, а затем сохраняет её в `%LOCALAPPDATA%\\vKOROBKU\\manual-games.json`; для неоднозначных папок доступно ручное уточнение названия, а вручную добавленную игру можно убрать из библиотеки без изменения файлов на диске;
- `FileTreeService` безопасно обходит дерево без перехода через reparse points.

### Analyzer

- тяжёлая инвентаризация, планирование выборки, измерение чтения и подсчёт физического размера выполняются вне UI-потока;
- анализ публикует текущий этап, общий процент и обработанный объём;
- инвентаризация логического и физического размера;
- стратифицированный выбор файлов с профилями 512 МБ, 1 ГБ, 2 ГБ и автоматическим лимитом;
- временная рабочая область на том же томе, где установлена игра: для системного диска — `%LOCALAPPDATA%\\vKOROBKU\\Analysis`, для другого диска — скрытый `<корень тома>\\vKOROBKU-Analysis`; осиротевшие рабочие каталоги старше часа удаляются в фоне при запуске, активные отмечаются PID процесса;
- оценка коэффициента сжатия;
- дисковый и декомпрессионный тест с чтением в обход системного кэша, отражающий скорость реального носителя игры;
- диапазон результата, категория влияния на загрузки и уровень доверия.

### Compression worker

Отдельный процесс `vKOROBKU.Worker` запускается через UAC и получает валидированное задание по двунаправленному named pipe с `PipeOptions.CurrentUserOnly`. Соединение защищено случайным одноразовым токеном. Секрет не передаётся в командной строке: App записывает его в скрытый одноразовый файл внутри `%LOCALAPPDATA%\\vKOROBKU\\WorkerAuth` с ACL только для текущего пользователя, Worker читает и сразу удаляет файл. UI постоянно работает без повышения прав.

Worker проверяет NTFS, запрещает корень диска, обнаруживает запущенные процессы игры, не переходит через reparse points и обрабатывает файлы безопасными пакетами через `compact.exe`. Поддерживаются прогресс, остановка, XPRESS4K/8K/16K, LZX и полная распаковка. Распаковка выполняется двумя проходами `compact.exe`: `/U /EXE` снимает WOF-сжатие (XPRESS/LZX), обычный `/U` — NTFS-сжатие. Отмена оставляет уже обработанные файлы в корректном состоянии.

Перед обработкой worker пропускает файлы, уже находящиеся в целевом состоянии (сжатые нужным алгоритмом при сжатии, несжатые при распаковке), и засчитывает их в прогресс. Благодаря этому прерванная операция продолжается с места остановки простым повторным запуском, а дообработка обновлённой игры затрагивает только новые файлы. Приложение работает в одном экземпляре (named mutex); при старте, если последняя операция — прерванное сжатие, предлагается продолжить его.

### Persistence

Состояние сжатия определяется через Windows WOF API и атрибут NTFS, а затем сохраняется в `%LOCALAPPDATA%\\vKOROBKU\\compression-status.json`. Это позволяет распознать как операции vKOROBKU, так и ранее сжатые сторонними инструментами игры. Статус присваивается по доле сжатых байт: от 85 % — «сжата», от 5 % — «сжата частично», иначе игра считается несжатой, чтобы единичные предсжатые файлы не блокировали оптимизацию.

Результаты анализа уже сохраняются атомарно в `%LOCALAPPDATA%\\vKOROBKU\\analysis-cache.json` и восстанавливаются по пути установки игры. Кэш содержит дату, размеры, выборку и прогнозы всех алгоритмов. Повторный анализ выполняется только по команде пользователя.

Технический журнал последних 200 операций хранится в `%LOCALAPPDATA%\\vKOROBKU\\operations.json`: состояние, прогресс, ошибки и факт прерывания. Раздел «Операции» показывает текущий анализ, сжатие или распаковку, позволяет остановить активную операцию и выводит историю. Одновременно выполняется только одна операция.

SQLite может быть добавлен позднее при росте объёма истории и настроек. Секреты и содержимое файлов не сохраняются. Телеметрия отсутствует.

## Направление зависимостей

UI → ViewModels → прикладные сервисы → Windows/файловая система. Модели не зависят от WPF. Системные операции скрыты за интерфейсами для модульного тестирования.

Основной пользовательский сценарий намеренно состоит из выбора игры и одной кнопки «Оптимизировать». Анализ, выбор безопасного алгоритма, журнал и проверка обновлений работают автоматически и не перегружают главный экран. Необязательный экспертный режим раскрывает точность анализа, результаты всех алгоритмов и ручной выбор; настройка запоминается между запусками.

## План этапов

1. Каркас UI, информация о ПК, Steam и ручная папка — готово.
2. Инвентаризация и безопасная выборка — готово.
3. Оценка сжатия XPRESS/LZX и доверительного диапазона — готово.
4. Бенчмарк чтения и понятная оценка влияния — готово.
5. Worker, реальное сжатие, прогресс, отмена и распаковка — готово, требуются полевые тесты.
6. Постоянный журнал операций и восстановление состояния после сбоя.
7. Кэш анализа — готово; SQLite для истории операций, полировка UI, тесты и GPL-релиз.
