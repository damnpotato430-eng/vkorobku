# Архитектура vKOROBKU

## Компоненты

### vKOROBKU.App

WPF-клиент на .NET 8 с MVVM. Отвечает за библиотеку игр, выбор режимов, визуализацию анализа, очередь и историю. Обычный запуск — `asInvoker`.

### Сканеры

- `SteamLibraryScanner` читает реестр, `libraryfolders.vdf` и `appmanifest_*.acf`;
- ручной источник принимает выбранный каталог, определяет игру по GOG-метаданным, `steam_appid.txt`, EXE и каталогу Steam, а затем сохраняет её в `%LOCALAPPDATA%\\vKOROBKU\\manual-games.json`;
- `FileTreeService` безопасно обходит дерево без перехода через reparse points.

### Analyzer

- инвентаризация логического и физического размера;
- стратифицированный выбор файлов с профилями 512 МБ, 1 ГБ, 2 ГБ и автоматическим лимитом;
- временная рабочая область;
- оценка коэффициента сжатия;
- дисковый и декомпрессионный тест с чтением в обход системного кэша;
- диапазон результата, категория влияния на загрузки и уровень доверия.

### Compression worker

Отдельный процесс `vKOROBKU.Worker` запускается через UAC и получает валидированное задание по двунаправленному named pipe. Соединение защищено случайным одноразовым токеном. UI постоянно работает без повышения прав.

Worker проверяет NTFS, запрещает корень диска, обнаруживает запущенные процессы игры, не переходит через reparse points и обрабатывает файлы безопасными пакетами через `compact.exe`. Поддерживаются прогресс, остановка, XPRESS4K/8K/16K, LZX и полная распаковка. Отмена оставляет уже обработанные файлы в корректном состоянии.

### Persistence

Состояние сжатия определяется через Windows WOF API и атрибут NTFS, а затем сохраняется в `%LOCALAPPDATA%\\vKOROBKU\\compression-status.json`. Это позволяет распознать как операции vKOROBKU, так и ранее сжатые сторонними инструментами игры.

Результаты анализа уже сохраняются атомарно в `%LOCALAPPDATA%\\vKOROBKU\\analysis-cache.json` и восстанавливаются по пути установки игры. Кэш содержит дату, размеры, выборку и прогнозы всех алгоритмов. Повторный анализ выполняется только по команде пользователя.

Технический журнал последних 200 операций хранится в `%LOCALAPPDATA%\\vKOROBKU\\operations.json`: состояние, прогресс, ошибки и факт прерывания. Он работает автоматически и не занимает место на главном экране.

SQLite может быть добавлен позднее при росте объёма истории и настроек. Секреты и содержимое файлов не сохраняются. Телеметрия отсутствует.

## Направление зависимостей

UI → ViewModels → прикладные сервисы → Windows/файловая система. Модели не зависят от WPF. Системные операции скрыты за интерфейсами для модульного тестирования.

Основной пользовательский сценарий намеренно состоит из выбора игры и одной кнопки «Оптимизировать». Анализ, выбор безопасного алгоритма, журнал и проверка обновлений работают автоматически и не перегружают главный экран. Необязательный экспертный режим раскрывает точность анализа, результаты всех алгоритмов и ручной выбор; настройка запоминается между запусками.

## План этапов

1. Каркас UI, информация о ПК, Steam и ручная папка — готово.
2. Инвентаризация и безопасная выборка — готово.
3. Оценка сжатия XPRESS/LZX и доверительного диапазона — готово.
4. Бенчмарк чтения и понятная оценка влияния — готово.
5. Worker, реальное сжатие, прогресс, отмена и распаковка — готово, требуются полевые тесты.
6. Постоянный журнал операций и восстановление состояния после сбоя.
7. Кэш анализа — готово; SQLite для истории операций, полировка UI, тесты и GPL-релиз.
